The biggest downloads on the homepage are not your content — they're Stripe (5+ scripts, ~1.9 MB), Zeffy, hCaptcha, reCAPTCHA and Google Pay — the entire donation/payment stack loading on every page whether or not anyone donates. On top of that, WordPress's editor scripts (block-editor.min.js 323 KB + components.min.js 260 KB) are loading on the public site.
Why it matters A 49-second largest paint on mobile means many visitors leave before the page draws. Page speed is also a Google ranking factor.
How to fix
  • Load the Zeffy/donation embed only on the Donate page (or on-click). If your donation embed is pasted into a header/footer/popup template, move it to the Donate page only.
  • Stop the editor scripts loading on the front end with the snippet below.
  • Add page caching + lazy-loading (WP Rocket or the host's cache).
✂ Fix pack — copy & paste
Paste into your child theme's functions.php (Appearance → Theme File Editor) or a Code Snippets plugin. Test the site after — if a front-end block stops rendering, remove the last two lines of the dequeue list.
PHP — functions.php
/* Digi2U audit fix #1 — stop editor + payment scripts loading site-wide */
add_action('wp_enqueue_scripts', function () {
    // WordPress editor code does not belong on the public site
    wp_dequeue_script('wp-block-editor');
    wp_dequeue_script('wp-editor');
    wp_dequeue_style('wp-block-editor');
    wp_dequeue_script('wp-components');
    wp_dequeue_style('wp-components');

    // Load donation/payment scripts ONLY on the Donate page (slug below)
    if (!is_page('donations-2')) {
        wp_dequeue_script('zeffy-embed');   // adjust handle to your donation plugin
        wp_dequeue_script('stripe');
        wp_dequeue_script('stripe-js');
    }
}, 100);
Script "handles" vary by plugin — if the payment scripts still load, a free plugin like Asset CleanUp shows every script per page with an off switch, no code needed.

#7Brand red fails contrast: #FF0000 text/buttons on white are below WCAG AAP2 · HIGHEffort: very low
Evidence axe-core: 11–16 "serious" contrast failures per page, all one cause — pure red #FF0000 on white (and white on red buttons) = 3.99:1; WCAG AA needs 4.5:1. Your design system already defines --d2u-red-dark: #D90000 which passes at 4.6:1.
How to fix Paste this into Appearance → Customize → Additional CSS (or ask Claude to apply it via the connector like the previous contrast fix):
✂ Fix pack — copy & paste CSS
CSS — Additional CSS
/* Digi2U audit fix #7 — WCAG AA: pure #FF0000 is 3.99:1 on white (needs 4.5:1).
   Shift interactive red to the brand's dark red. Large display headlines
   (36px+) may keep pure red — large text only needs 3:1. */
html:root { --d2u-red: #D90000; }
The html:root selector out-ranks the page's own :root definition, so this wins everywhere the variable is used.
